  The reframing in one line: the agent doesn't produce facts — it routes between

  Why this works:

  Every numeric field in the final report can be made to trace back to a tool
  call. The LLM never writes a calorie count; it calls

  function returns major_discrepancy. The LLM never picks a candidate by feel;
  it calls select_best_candidate(...) which returns the top-scored entry with a
  deterministic tie-breaker (e.g., lowest fdcId).

  Stochasticity collapses into:
  - Which tools the agent called — order, redundancy, occasional skip. Doesn't
  affect output values.
  - Tool-call timing — irrelevant.

  The output values themselves become deterministic as long as:
  1. No tool returns prose that gets propagated to the report.
  2. All arithmetic is in tools, not in the LLM's head.
  3. The output schema (output_type=ItemVerification) forces every field to be
  structured.
  4. Ties and ambiguity have deterministic tie-breakers inside the tools.
  5. API responses are pinned via FileCache (so re-runs hit the cache, not the
  live API).

  during a code review and say "this is the contract." Each tool gets its own
  file (matches your "max separation" rule), its own type signature, its own

     Nutrition Verification Agent — Implementation Plan

     Context

     SNAQ (nutrition app) take-home: build an "agentic system" that takes
     food_items.json (11 items with nutrition_per_100g) and verifies whether the
      values are correct, flagging discrepancies and proposing corrections.
     Deliverables: runnable code, README, the produced output, and the AI
     conversation log.

     The architectural choice driving this plan: the LLM agent orchestrates tool
      calls; tools produce all facts. Numeric values, candidate selection,
     verdicts, and human-readable summaries all trace back to a tool call —
     never to LLM prose. The LLM is a router; the tools are the truth. Output
     guardrails re-check the LLM's claims against pure math (Atwater equation,
     schema), so hallucinations cannot survive into the report. This satisfies
     "agentic system" without sacrificing reproducibility — re-running the
     verifier on the same input yields the same verification_report.json (modulo
      trace order, which is non-load-bearing).

     Why not a pure deterministic pipeline: the brief explicitly asks for an
     agentic system with tool use. We honor that — but discipline the agent so
     its non-determinism is confined to which tools it calls (irrelevant to
     output values), not what numbers it produces.

     Why not an LLM-heavy agent: determinism is a feature in a verification
     system. Source disagreement is real (the input file likely contains at
     least one planted error — Fage Total 0% protein listed at 5.5g/100g;
     real-world value is ~10g/100g), so the system needs structured comparison
     and graceful uncertainty handling — neither requires LLM "judgment" inside
     the verification path.

     Where LLMs earn their keep besides routing: the eval layer. verify eval
     runs an LLM-as-judge against a small hand-curated ground_truth.json. This
     is the bonus deliverable, scoped so further eval steps (regression, trace
     replay, full eval rig) plug in via the same PipelineStep abstraction.

     Mirror ../brella-mass-outbound conventions: uv + pyproject.toml, ruff
     (Google docstrings, 88 cols, double quotes), mypy strict, Python 3.14+, ABC
      ports with @abstractmethod, constructor-injected adapters,
     Bootstrap.build() composition root, structlog via LoggerPort, Typer + Rich
     CLI, pytest with fakes (no mocks). One deliberate deviation: domain models
     are Pydantic v2 BaseModel (not dataclasses) — the project is small, the
     output is JSON-first via model_dump_json(), and Pydantic plugs straight
     into the OpenAI Agents SDK output_type hook.

     Team-friendly file separation. Every model, port, adapter, tool, and
     pipeline step lives in its own file. Once contracts are frozen (Phase 1),
     independent teammates can implement adapters/tools/steps in parallel
     without merge conflicts.

     TDD throughout. Every teammate writes tests first (one test file per source
      file, one test per branch/case), runs them red, implements, runs them
     green, refactors. Granular: each tool has its own test file with cases for
     nominal, empty, malformed, and edge inputs. Adapters are tested with respx
     against pinned response fixtures. The agent layer is tested with
     set_tracing_disabled(True) and stubbed tool responses. The orchestrator
     (me) verifies the full test matrix is green before integration.

     Teammate mode (TeamCreate). I act as orchestrator/PM. Eight teammates run
     in parallel, each owning one domain end-to-end (adapter + tools + fakes +
     tests). Coordination happens via the shared task list and named-message
     DMs. Two of the eight are special: docs-oracle is consulted by every other
     teammate before implementing (no agent writes API code from memory), and
     credentials-runner drives Chrome via chrome-mcp with the user's help to
     provision API keys and populate .env. Drift control: the orchestrator
     reviews each teammate's test list before they implement, and rejects PRs
     that exceed scope.

     Eval pipeline (opt-in via `verify eval`):

     Determinism guarantees

     1. All tool functions return structured Pydantic data — never prose that
     gets propagated to the report.
     2. All arithmetic lives in tools; the LLM never computes a number itself.
     3. select_best_candidate uses a fixed scoring formula (token overlap on
     name + brand exact match + USDA dataType priority + OFF
     completeness/popularity_key) with a deterministic tie-break (lowest fdcId
     for USDA, lowest code for OFF).
     4. format_human_summary is a string template, never an LLM call.
     5. Output schema (ItemVerification) requires fields that only specific
     tools produce — the LLM cannot ship a verdict without calling
     verdict_from_deltas.
     6. FileCache pins API responses across runs so even the underlying source
     data is stable.
     7. Output guardrail re-runs check_atwater_consistency on the agent's
     synthesized output and tripwires if the verdict contradicts pure math.

⏺ Three honest reasons the tests missed it:

  1. No test ever calls Bootstrap.build(). Each adapter has its own unit tests
  with the real constructor, but the composition root — where the wires actually
   meet — is untested. Pipeline-cli-domain's pipeline-step tests use
  FakeVerifierAgent/FakeJudgeAgent, which sidestep Bootstrap.build() entirely.
  Classic hexagonal blind spot.
  2. Mypy was muzzled on the infrastructure layer. Remember in Phase 1 I added
  this to pyproject.toml:
  [[tool.mypy.overrides]]
  module = ["snaq_verify.infrastructure.*"]
  ignore_missing_imports = true
  2. That was a temporary override so the bootstrap could lazy-import adapters
  before they existed. I never removed it after Phase 2 landed. With it active,
  mypy treats those modules as Any-typed, so FileCache(cache_dir=, ttl_seconds=,
   logger=) looks fine to the type checker — every kwarg silently passes.
  Without that override, mypy would have failed at mypy src/.
  3. Fakes' signatures aren't required to match real adapters'. The fakes
  implement the port interface, which is the only contract the unit tests assert
   against. Constructor args of the real adapter are pure implementation detail
  to those tests. So a real adapter can drift its constructor and no fake-based
  test will notice.

  Three fixes, in order of leverage:
  - Remove the mypy override now that adapters exist. Permanent guardrail;
  surfaces this exact class of bug at make typecheck time.
  - Add a Bootstrap.build() smoke test that runs in unit tests with a stub
  Settings (env-vars set to dummy values, no network calls). Catches
  construction-level mismatches without needing real APIs.
  - Long-term: a contract test per port that runs the same scenario through both
   the fake and the real adapter (with a recorded HTTP cassette) — but that's
  overkill for a take-home.

  I'll do the first two as part of Phase 3 cleanup once pipeline-cli-domain
  finishes the bootstrap fix.
